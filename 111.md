> 当前版本：v1.0

## 1. 项目总览

### 1.1 项目一句话介绍

本项目是一个面向电商买家的智能客服系统。用户可以注册登录后，通过网页端与 AI 客服对话，也可以在订单、售后、个人信息等页面中完成传统表单式操作。后端提供 FastAPI REST API，数据持久化使用 MySQL，Agent 层使用 LangGraph 编排 LLM 与业务工具，前端使用 React + Vite 构建单页应用。

### 1.2 项目目标

项目的核心目标不是直接做一个商用客服系统，而是通过一个完整业务场景练习以下能力：

- Python Web 后端开发：FastAPI 路由、依赖注入、统一异常处理、JWT 鉴权。
- 数据库建模：用户、订单、物流、售后、对话、消息、转人工、会话记忆等表结构。
- 异步 ORM：SQLAlchemy 2.0 async、AsyncSession、异步查询、事务提交。
- 前后端分离：React 页面、路由守卫、API 封装、Vite 代理。
- AI Agent 工程化：LangGraph 状态图、工具调用、结构化 ToolResult、任务状态机。
- 会话记忆：短期历史、摘要压缩、任务状态持久化、偏好和事实抽取。
- 测试意识：对 Agent 工具、状态机、图辅助函数、记忆服务进行可重复测试。

### 1.3 适用场景

本项目适合用于：

- 个人简历项目展示。
- 学习 FastAPI + React 全栈开发。
- 学习 LLM 工具调用和 LangGraph 编排。
- 演示“传统业务系统如何接入智能客服”。
- 作为更复杂客服系统、订单系统、售后系统的原型。

不适合直接用于：

- 真实生产环境。
- 大规模并发客服系统。
- 涉及真实支付、真实退款、真实物流回调的业务。
- 对安全合规、审计、权限隔离要求很高的企业场景。

## 3. 系统架构

### 3.1 总体架构

```text
浏览器 / 用户
    |
    | HTTP
    v
React + Vite 前端
    |
    | /api 代理或直接请求
    v
FastAPI 后端
    |
    | SQLAlchemy AsyncSession
    v
MySQL 数据库

FastAPI 后端
    |
    | POST /api/chat/message
    v
Chat Service
    |
    | run_agent(...)
    v
LangGraph Agent
    |
    | Tool Calling
    v
订单 / 物流 / 售后 Service

LangGraph Agent
    |
    | OpenAI-compatible SDK
    v
LLM 服务
```

### 3.2 分层设计

后端采用清晰的分层结构：

| 层级 | 目录 | 职责 |
|---|---|---|
| 入口层 | `app/main.py` | 创建 FastAPI app、注册中间件、注册路由、初始化数据库、统一异常处理 |
| 路由层 | `app/routers/` | 定义 HTTP API，解析请求，调用 service，不写复杂业务逻辑 |
| Schema 层 | `app/schemas/` | 定义 Pydantic 请求体和响应体 |
| Service 层 | `app/services/` | 业务逻辑、权限校验、数据库读写、事务提交 |
| Model 层 | `app/models/` | SQLAlchemy 表结构定义 |
| Agent 层 | `app/agent/` | LangGraph 编排、工具定义、状态机、提示词、结构化结果 |
| 测试层 | `tests/` | 单元测试和评估脚本 |

### 3.3 关键请求链路

#### 3.3.1 普通 REST API 链路

以查询订单详情为例：

```text
前端 Orders.jsx
  -> frontend/src/api/index.js:getOrderDetail(id)
  -> GET /api/orders/{id}
  -> app/routers/orders.py:get_order
  -> Depends(get_current_user) 解析 JWT
  -> app/services/order_service.py:get_order_detail
  -> MySQL 查询 orders + order_items
  -> OrderResponse
  -> 前端渲染订单详情
```

#### 3.3.2 Agent 对话链路

以用户说“查一下 202605280002 的物流”为例：

```text
前端 Chat.jsx
  -> sendMessage(conversationId, content)
  -> POST /api/chat/message
  -> chat_service.process_agent_message
  -> 保存用户消息 messages(role=user)
  -> 读取最近 10 条消息作为上下文
  -> run_agent(...)
  -> LangGraph: load_memory
  -> LangGraph: call_llm
  -> LLM 选择 query_logistics 工具
  -> LangGraph: execute_tool
  -> ecommerce.execute_tool
  -> order_service.get_order_logistics
  -> ToolResult 写入 ToolMessage
  -> LangGraph: call_llm 再次生成自然语言回复
  -> 保存 agent 消息 messages(role=agent)
  -> 保存 task_state
  -> 调度后台记忆摘要任务
  -> 返回 reply 给前端
```

## 5. 环境配置

### 5.1 后端环境要求

- Python 3.9 或更高版本。
- MySQL 8.0 或更高版本。
- 推荐使用虚拟环境。

### 5.2 前端环境要求

- Node.js 18 或更高版本。
- npm。

### 5.3 后端依赖安装

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 5.4 前端依赖安装

```bash
cd frontend
npm install
```

### 5.5 `.env` 配置

项目通过 `app/config.py` 加载 `.env`。可以从 `.env.example` 复制一份：

```bash
cp .env.example .env
```

推荐配置：

```env
DATABASE_URL=mysql+aiomysql://root@localhost:3306/ecommerce_cs

SECRET_KEY=your-secret-key-change-in-production
ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=30

OPENAI_API_KEY=sk-xxx
OPENAI_BASE_URL=https://api.openai.com/v1
LLM_MODEL=gpt-4o-mini

LLM_SUMMARY_API_KEY=sk-xxx
LLM_SUMMARY_BASE_URL=https://api.openai.com/v1
LLM_SUMMARY_MODEL=gpt-4o-mini
```

说明：

- `DATABASE_URL` 使用异步驱动 `mysql+aiomysql`。
- `SECRET_KEY` 在生产环境必须替换成足够随机的密钥。
- `OPENAI_BASE_URL` 可配置为任何兼容 OpenAI Chat Completions API 的服务地址。
- 摘要模型配置为空时，默认复用主链路 LLM 配置。

## 6. 启动流程

### 6.1 创建数据库

```sql
CREATE DATABASE IF NOT EXISTS ecommerce_cs
  DEFAULT CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;
```

或者使用命令行：

```bash
mysql -u root -e "CREATE DATABASE IF NOT EXISTS ecommerce_cs DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
```

### 6.2 启动后端

```bash
./run.sh
```

`run.sh` 会执行：

```bash
unset DATABASE_URL
exec uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

后端启动后：

- API 根地址：http://localhost:8000
- Swagger 文档：http://localhost:8000/docs
- OpenAPI JSON：http://localhost:8000/openapi.json

### 6.3 初始化种子数据

```bash
python3 seed_data.py
```

种子数据会清空旧数据并插入：

- 测试用户：`buyer1 / 123456`
- 4 条订单，覆盖 delivered、shipped、paid、pending 状态。
- 每条订单的商品明细。
- 每条订单的物流数据。
- 1 条售后记录。

### 6.4 启动前端

```bash
cd frontend
npm run dev
```

默认访问：

- 前端地址：http://localhost:5173
- 登录页：http://localhost:5173/login
- 聊天页：http://localhost:5173/chat

Vite 已配置代理：

```js
server: {
  proxy: {
    '/api': {
      target: 'http://localhost:8000',
      changeOrigin: true,
    },
  },
}
```

因此前端代码只需要请求 `/api/...`，开发服务器会自动转发到后端。

## 7. 数据库设计

### 7.1 表总览

当前系统共有 9 张核心表：

```text
用户体系:
  users

订单体系:
  orders
  order_items
  logistics

售后体系:
  after_sales

对话体系:
  conversations
  messages
  transfer_logs
  conversation_memories
```

### 7.2 表关系

```text
users 1 ── N orders
orders 1 ── N order_items
orders 1 ── 1 logistics

users 1 ── N after_sales
orders 1 ── N after_sales

users 1 ── N conversations
conversations 1 ── N messages
conversations 1 ── N transfer_logs
conversations 1 ── N conversation_memories
```

### 7.3 `users` 用户表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK, auto increment | 用户 ID |
| username | varchar(50) | unique, index | 用户名 |
| password_hash | varchar(255) | not null | 密码哈希 |
| phone | varchar(20) | nullable | 手机号 |
| default_address | varchar(255) | nullable | 默认收货地址 |
| created_at | datetime | server_default now | 注册时间 |
| updated_at | datetime | on update now | 更新时间 |

密码使用 `pbkdf2_sha256` 哈希，不保存明文密码。

### 7.4 `orders` 订单表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK, auto increment | 数据库订单 ID |
| user_id | int | FK -> users.id, index | 所属用户 |
| order_no | varchar(64) | unique, index | 业务订单号 |
| status | varchar(20) | default pending | 订单状态 |
| total_amount | decimal(10,2) | not null | 总金额 |
| shipping_address | varchar(255) | nullable | 收货地址快照 |
| created_at | datetime | server_default now | 下单时间 |
| updated_at | datetime | on update now | 更新时间 |

订单状态：

| 状态 | 中文含义 | 是否可取消 |
|---|---|---|
| pending | 待处理 / 待付款 | 是 |
| paid | 已付款 | 是 |
| shipped | 已发货 | 否 |
| delivered | 已签收 | 否 |
| cancelled | 已取消 | 否 |

### 7.5 `order_items` 订单商品表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK | 商品项 ID |
| order_id | int | FK -> orders.id, index | 所属订单 |
| product_name | varchar(200) | not null | 商品名称 |
| quantity | int | not null | 数量 |
| price | decimal(10,2) | not null | 单价 |

### 7.6 `logistics` 物流表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK | 物流 ID |
| order_id | int | FK -> orders.id, index | 所属订单 |
| company | varchar(50) | nullable | 快递公司 |
| tracking_no | varchar(100) | nullable | 快递单号 |
| status | varchar(50) | default pending | 物流状态 |
| updated_at | datetime | on update now | 最新更新时间 |

当前种子数据中的物流状态包括：

- 已签收
- 运输中
- 待发货
- pending

### 7.7 `after_sales` 售后表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK | 售后 ID |
| order_id | int | FK -> orders.id, index | 关联订单 |
| user_id | int | FK -> users.id, index | 用户 ID |
| type | varchar(20) | not null | 售后类型 |
| reason | text | not null | 售后原因 |
| status | varchar(20) | default pending | 售后状态 |
| created_at | datetime | server_default now | 申请时间 |
| updated_at | datetime | on update now | 更新时间 |

售后类型：

| 类型 | 中文含义 |
|---|---|
| return | 退货 |
| refund | 退款 |
| exchange | 换货 |

售后状态：

| 状态 | 中文含义 |
|---|---|
| pending | 待审核 |
| approved | 已通过 |
| rejected | 已拒绝 |
| completed | 已完成 |

### 7.8 `conversations` 会话表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK | 会话 ID |
| user_id | int | FK -> users.id, index | 用户 ID |
| status | varchar(20) | default active | 会话状态 |
| created_at | datetime | server_default now | 创建时间 |
| updated_at | datetime | on update now | 最后更新时间 |

当前会话策略：

- 一个用户只保留一个 active 会话。
- `GET /api/chat/sessions` 返回这个 active 会话。
- `POST /api/chat/session` 获取或创建 active 会话。

### 7.9 `messages` 消息表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK | 消息 ID |
| conversation_id | int | FK -> conversations.id, index | 所属会话 |
| role | varchar(20) | not null | user / agent / system |
| content | text | not null | 消息内容 |
| created_at | datetime | server_default now | 发送时间 |

Agent 主链路中会保存两类消息：

- 用户输入：`role=user`
- Agent 回复：`role=agent`

前端本地也可能展示 `system` 类型错误消息，但后端主流程目前主要保存 user 和 agent。

### 7.10 `transfer_logs` 转人工记录表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK | 转人工记录 ID |
| conversation_id | int | FK -> conversations.id, index | 会话 ID |
| user_id | int | FK -> users.id, index | 用户 ID |
| reason | text | not null | 转人工原因 |
| status | varchar(20) | default pending | 处理状态 |
| created_at | datetime | server_default now | 转接时间 |

该表目前是预留能力，现有 Agent 工具列表尚未暴露 `transfer_to_human` 工具。

### 7.11 `conversation_memories` 会话记忆表

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | int | PK | 记忆 ID |
| conversation_id | int | FK -> conversations.id, index | 会话 ID |
| user_id | int | FK -> users.id, index | 用户 ID |
| memory_type | varchar(20) | index | 记忆类型 |
| content | text | not null | 记忆内容 |
| created_at | datetime | server_default now | 创建时间 |
| updated_at | datetime | on update now | 更新时间 |

唯一约束：

```text
conversation_id + memory_type 唯一
```

这意味着同一会话下，同一种记忆只保留最新版本。

记忆类型：

| 类型 | 说明 |
|---|---|
| summary | LLM 压缩后的会话摘要 |
| task_state | 当前任务状态 JSON |
| fact | 从对话中抽取出的用户事实 |
| preference | 从对话中抽取出的用户偏好 |
| summary_cursor | 摘要压缩已经处理到的 message id |
| rule_cursor | 规则抽取已经处理到的 message id |

## 10. 认证与权限设计

### 10.1 JWT 生成

`create_access_token(user_id)` 会生成以下 payload：

```json
{
  "sub": "1",
  "exp": "<过期时间>"
}
```

特点：

- `sub` 保存用户 ID 字符串。
- 过期时间由 `ACCESS_TOKEN_EXPIRE_MINUTES` 控制。
- 默认算法为 HS256。

### 10.2 当前用户解析

`get_current_user` 是 FastAPI 依赖：

```text
HTTP Authorization Header
  -> HTTPBearer
  -> jwt.decode
  -> payload.sub
  -> db.get(User, user_id)
  -> 返回 User
```

### 10.3 数据归属校验

订单、售后、会话均通过 `current_user.id` 限制访问：

- 查询订单：`Order.id == order_id` 且 `Order.user_id == current_user.id`
- 查询售后：`AfterSale.id == after_sale_id` 且 `AfterSale.user_id == current_user.id`
- 查询会话：`Conversation.id == conversation_id` 且 `Conversation.user_id == current_user.id`

这可以防止用户通过猜 ID 访问他人数据。

## 11. Agent 设计

### 11.1 Agent 能力范围

当前 Agent 支持：

- 查询当前用户订单列表。
- 查询指定订单详情。
- 查询指定订单物流。
- 提交售后申请。
- 根据工具结果生成自然语言回复。
- 读取会话记忆、任务状态、用户偏好和用户事实。
- 生成结构化 `TaskState`。
- 防止工具调用死循环。

当前 Agent 暂未直接支持：

- 真正的人工客服接入。
- 修改订单地址。
- 主动取消订单工具。
- 查询售后详情工具。
- 多用户客服工作台。
- 流式输出。

### 11.2 LangGraph 图结构

```text
START
  |
  v
load_memory
  |
  v
call_llm
  |
  +-- 如果有 tool_calls 且未达到上限 --> execute_tool
  |                                      |
  |                                      v
  |                                  call_llm
  |
  +-- 如果有 tool_calls 但达到上限 --> finalize_tool_limit --> END
  |
  +-- 如果无 tool_calls -------------> END
```

### 11.3 State 字段

`AgentState` 定义在 `app/agent/schemas/state.py`。

| 字段 | 说明 |
|---|---|
| messages | LangChain 消息列表，包含用户、AI、工具消息 |
| user_id | 当前用户 ID |
| conversation_id | 当前会话 ID |
| db | 当前请求的 AsyncSession |
| memory | load_memory 加载出的记忆字典 |
| task_state | 当前轮生成的任务状态 |
| tool_iterations | 当前轮工具调用轮数 |

### 11.4 节点职责

#### load_memory

职责：

- 从 `conversation_memories` 表加载当前会话记忆。
- 按 `memory_type` 组装为 dict。
- 写入 `state.memory`。

该节点不写数据库，避免图执行过程中产生额外副作用。

#### call_llm

职责：

- 组装 system prompt。
- 将会话记忆注入 prompt。
- 将 LangChain messages 转为 OpenAI API messages。
- 调用 LLM。
- 如果模型返回工具调用，转为 AIMessage.tool_calls。
- 如果模型返回自然语言，则生成 task_state。

#### execute_tool

职责：

- 遍历最新 AIMessage 的 tool_calls。
- 调用 `app.agent.tools.execute_tool`。
- 将 ToolResult 序列化为 ToolMessage。
- 根据工具结果更新 task_state。
- 增加 tool_iterations。

#### finalize_tool_limit

职责：

- 当工具调用轮数达到上限时，返回明确的用户可见回复。
- 将任务状态标记为 failed/error。
- 避免 Agent 因模型反复调用工具而无限循环。

### 11.5 工具列表

工具定义在 `app/agent/tools/ecommerce.py`。

| 工具 | 参数 | 说明 |
|---|---|---|
| query_orders | 无 | 查询当前用户订单列表 |
| query_order_detail | order_id | 查询订单详情 |
| query_logistics | order_id | 查询订单物流 |
| submit_after_sale | order_id, sale_type, reason | 提交售后申请 |

`order_id` 支持两种输入：

- 数据库数字 ID，例如 `1`。
- 业务订单号，例如 `202605280001`。

解析策略：

1. 优先按 `order_no` 查询。
2. 找不到时尝试转为 int，按数据库 ID 查询。
3. 两者都找不到则返回工具错误。

### 11.6 ToolResult 结构

工具执行结果统一使用 `ToolResult`：

```json
{
  "ok": true,
  "tool": "query_logistics",
  "message": "订单 1 物流：...",
  "data": {
    "order_id": 1,
    "status": "运输中",
    "status_label": "运输中"
  },
  "task_patch": {
    "stage": "completed",
    "intent": "query_logistics",
    "status": "done",
    "order_id": 1
  }
}
```

这样设计的好处：

- `message` 面向 LLM，便于生成自然语言回复。
- `data` 面向后端调试和后续扩展。
- `task_patch` 面向任务状态机。
- `error_code` 可表达工具失败原因。

### 11.7 Prompt 规则

系统 Prompt 位于 `app/agent/prompts.py`。

核心规则：

- 只处理电商客服相关问题。
- 用户闲聊、问候、感谢时直接回复，不调用工具。
- 查订单但没指定订单时，先调用 `query_orders`。
- 查物流但没指定订单时，先调用 `query_orders`。
- 提交售后必须具备 order_id、sale_type、reason。
- 工具失败后不要反复调用同一个工具。
- 不要编造订单状态、金额、物流、售后编号。
- 记忆仅作为参考，永远优先响应用户最新消息。

### 11.8 工具调用上限

`MAX_TOOL_ITERATIONS = 4`。

原因：

- 防止模型不断调用工具形成循环。
- 避免单次请求耗时过长。
- 出现复杂情况时，先让用户补充更明确的信息。

达到上限时会返回：

```text
抱歉，这次查询需要的工具步骤过多，我先暂停处理。请您换一种更具体的说法，或提供订单号后再试。
```

## 12. 任务状态机

### 12.1 TaskState 结构

`TaskState` 定义在 `app/agent/schemas/task_state.py`：

```json
{
  "version": 1,
  "stage": "completed",
  "intent": "query_logistics",
  "status": "done",
  "order_id": 1,
  "customer_id": 7,
  "confidence": 0.82,
  "next_action": "reply_user",
  "updated_at": "2026-06-05T10:00:00Z"

```

### 12.2 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| version | int | schema 版本 |
| stage | enum | 当前任务阶段 |
| intent | enum | 用户意图 |
| status | enum | 任务整体状态 |
| order_id | int/null | 当前关联订单 ID |
| customer_id | int/null | 当前用户 ID |
| confidence | float | 启发式分数，0-1 |
| next_action | enum | 下一步建议动作 |
| updated_at | datetime | 更新时间 |

### 12.3 stage 枚举

| 值 | 说明 |
|---|---|
| new | 新任务 |
| awaiting_order_id | 等待用户提供订单 |
| awaiting_confirmation | 等待用户确认 |
| processing | 正在处理 |
| completed | 已完成 |
| failed | 已失败 |

### 12.4 intent 枚举

| 值 | 说明 |
|---|---|
| query_order_status | 查询订单状态或订单详情 |
| query_logistics | 查询物流 |
| query_after_sale | 查询售后 |
| submit_after_sale | 提交售后 |
| cancel_order | 取消订单 |
| transfer_human | 转人工 |
| other | 其他 |

### 12.5 status 枚举

| 值 | 说明 |
|---|---|
| pending | 待处理 |
| in_progress | 处理中 |
| done | 已完成 |
| error | 出错 |

### 12.6 next_action 枚举

| 值 | 说明 |
|---|---|
| ask_user_for_order_id | 询问订单号 |
| confirm_user_intent | 确认意图 |
| call_backend_api | 调用后端 API |
| reply_user | 回复用户 |
| transfer_to_human | 转人工 |
| stop | 停止 |

### 12.7 状态更新规则

状态机核心函数：

```python
reduce_task_state(...)
```

主要规则：

- 如果最新工具结果包含 `task_patch`，优先使用工具结果修补状态。
- 如果工具失败，状态变为 `failed/error`。
- 如果工具成功，状态变为 `completed/done`。
- 如果有工具调用但没有结构化工具结果，状态暂定为 `processing/in_progress`。
- 如果用户消息包含“订单”或“物流”，但没有工具结果，状态进入 `awaiting_order_id`。
- 其他情况归为 `new/other/pending`。

### 12.8 confidence 说明

`confidence` 当前不是模型真实概率，而是启发式分数。

计算公式：

```text
score = model_signal * 0.4 + rule_signal * 0.3 + context_signal * 0.3
```

所有输入都会被夹紧到 `[0.0, 1.0]`。

## 13. 会话记忆系统

### 13.1 记忆系统目标

会话记忆解决三个问题：

- 对话变长后，不能每次都把全部历史传给模型。
- 用户可能在多轮中省略上下文，例如“刚才那个订单”。
- 系统需要保存任务状态，便于下一轮继续。

### 13.2 记忆类型

| 类型 | 生成方式 | 用途 |
|---|---|---|
| summary | LLM 摘要压缩 | 保留对话主题、目标、进展、下一步 |
| task_state | 状态机生成 | 保存结构化任务状态 |
| preference | 规则抽取 | 保存用户表达偏好，如“回答简洁一点” |
| fact | 规则抽取 | 保存用户事实，如默认地址 |
| summary_cursor | 系统写入 | 记录摘要处理到哪条消息 |
| rule_cursor | 系统写入 | 记录规则抽取处理到哪条消息 |

### 13.3 主链路与后台任务

用户发送消息后：

```text
保存用户消息
  -> 运行 Agent
  -> 保存 Agent 回复
  -> 主链路保存 task_state
  -> 调度后台 save_memory_background
  -> 立即返回给用户
```

后台任务执行：

```text
保存 task_state 兜底
  -> 根据 rule_cursor 抽取 preference/fact
  -> 根据 summary_cursor 读取新增消息
  -> 调用摘要模型压缩 summary
  -> 保存 summary 和 summary_cursor
```

设计原则：

- 用户回复优先返回。
- 摘要失败不影响主链路。
- 记忆写入失败记录日志，不让用户请求失败。
- `memory_service` 不依赖 `chat_service`，避免循环依赖。

### 13.4 摘要 Prompt

摘要要求：

- 2-4 句话。
- 只保留对后续对话有用的信息。
- 包含用户当前在做什么、想达成什么、进展、下一步。
- 不包含订单号、物流单号、原始工具数据。
- 不输出标签或解释。

### 13.5 规则记忆抽取

当前规则：

- 用户说“简洁 / 简短 / 短一点 / 少一点”：
  - 保存 preference：`用户希望回答简洁。`
- 用户说“详细 / 展开说 / 多解释”：
  - 保存 preference：`用户希望回答更详细。`
- 用户说“默认地址是...”或“收货地址是...”：
  - 保存 fact：`用户的默认地址是...。`

## 14. 前端设计

### 14.1 页面结构

| 页面 | 路径 | 说明 |
|---|---|---|
| 登录 | `/login` | 用户输入用户名密码登录 |
| 注册 | `/register` | 创建新用户 |
| 聊天 | `/chat` | 智能客服主页面 |
| 订单 | `/orders` | 查看、创建、取消订单，查看详情和物流 |
| 售后 | `/after-sales` | 查看售后记录，提交售后申请 |
| 个人信息 | `/profile` | 查看和修改手机号、地址 |

### 14.2 路由守卫

`App.jsx` 中定义两种守卫：

- `ProtectedRoute`：必须登录才能访问。
- `GuestRoute`：登录 / 注册页只允许未登录用户访问。

路由规则：

```text
/login          未登录可访问，已登录跳 /chat
/register       未登录可访问，已登录跳 /chat
/chat           登录后访问
/orders         登录后访问
/after-sales    登录后访问
/profile        登录后访问
*               默认跳 /chat
```

### 14.3 登录态管理

`AuthContext.jsx` 负责：

- 启动时从 localStorage 读取 token。
- 调用 `/api/users/me` 验证 token。
- 保存当前用户信息。
- 登录成功后写入 token。
- 退出时清除 token 和 user。

### 14.4 API 封装

`frontend/src/api/index.js` 提供：

- 自动加 `Content-Type: application/json`。
- 自动加 `Authorization: Bearer <token>`。
- 统一解析 JSON。
- 统一处理 401，清除 token 并跳转登录页。
- 将后端 `message` 抛为 Error。

### 14.5 聊天页面

`Chat.jsx` 主要功能：

- 页面加载时创建或获取会话。
- 拉取历史消息。
- 没有历史时展示欢迎语。
- 输入框支持 Enter 发送、Shift+Enter 换行。
- 发送时先乐观渲染用户消息。
- 等待 Agent 回复时显示 typing 状态。
- Agent 回复支持简单 Markdown 表格和加粗渲染。
- 底部提供快捷入口：我的订单、售后服务、个人信息。

### 14.6 订单页面

`Orders.jsx` 主要功能：

- 分页加载订单。
- 展示订单号、日期、状态、金额。
- 点击订单展开详情。
- 同时加载订单详情和物流。
- pending / paid 订单支持取消。
- 支持创建测试订单。
- 支持上一页 / 下一页。

### 14.7 售后页面

`AfterSales.jsx` 主要功能：

- 加载售后记录。
- 加载可申请售后的订单。
- 过滤订单状态为 paid / shipped / delivered。
- 支持选择订单、售后类型、填写原因并提交。
- 展示售后状态和申请时间。

### 14.8 个人信息页面

`Profile.jsx` 主要功能：

- 展示用户名、手机号、地址、注册时间。
- 支持编辑手机号和地址。
- 保存成功后重新拉取用户信息。
- 支持退出登录。

## 16. 常见开发任务

### 16.1 新增一个普通 API

推荐步骤：

1. 在 `app/schemas/` 中定义请求和响应模型。
2. 在 `app/services/` 中实现业务逻辑。
3. 在 `app/routers/` 中定义路由。
4. 在 `app/main.py` 中确认 router 已 include。
5. 在前端 `frontend/src/api/index.js` 中封装请求。
6. 在页面中调用 API。
7. 补充测试。

### 16.2 新增一个 Agent 工具

推荐步骤：

1. 在 `app/agent/tools/ecommerce.py` 中定义 `@tool` 函数声明。
2. 在 `_execute_tool` 中实现真实执行逻辑。
3. 返回结构化 `ToolResult`。
4. 在 `_intent_for_tool` 和 `_TOOL_INTENT_MAP` 中补充映射。
5. 将工具加入 `AGENT_TOOLS`。
6. 更新 `app/agent/prompts.py` 的工具规则。
7. 补充 `tests/test_agent_tools.py` 和状态机测试。

### 16.3 新增一种记忆类型

推荐步骤：

1. 在 `MemoryType` 枚举中添加类型。
2. 在 `ConversationMemory.memory_type` 注释中更新说明。
3. 在 `_format_memory_for_prompt` 中决定是否注入 prompt。
4. 在 `save_memory_background` 中决定写入时机。
5. 为读取、写入、prompt 拼装补测试。

### 16.4 新增前端页面

推荐步骤：

1. 在 `frontend/src/pages/` 新建页面组件。
2. 在 `frontend/src/App.jsx` 添加路由。
3. 如果需要登录，放入 `ProtectedRoute`。
4. 在 `frontend/src/api/index.js` 中封装对应接口。
5. 在已有页面中添加跳转入口。
6. 在 `frontend/src/index.css` 中复用现有样式。

## 17. 当前限制与改进方向

### 17.1 后端限制

- 目前没有 Alembic migration，开发阶段依赖 `Base.metadata.create_all`。
- 没有 refresh token，`/refresh` 只是用当前 access token 换新 access token。
- 没有角色权限系统，只有买家用户。
- 没有真实支付、库存、商家审核。
- 没有订单状态自动流转任务。
- 售后申请没有状态审核接口。

### 17.2 Agent 限制

- 依赖外部 LLM，可用性受 API Key、网络、模型质量影响。
- 工具列表较少，尚未覆盖取消订单、查询售后详情、转人工。
- `confidence` 是启发式分数，不是严格概率。
- 没有流式输出，长回复需要等待完整返回。
- 没有工具调用审计页面。
- 没有 prompt 版本管理。

### 17.3 前端限制

- 没有 TypeScript。
- 没有组件库。
- 没有端到端测试。
- token 存在 localStorage，简单但不适合高安全要求场景。
- 错误处理以页面提示为主，没有全局 toast 系统。

### 17.4 数据库限制

- 外键关系没有通过 ORM relationship 完整建模。
- 没有软删除字段。
- 没有审计日志。
- 没有索引优化分析。
- 没有迁移脚本。

## 19. 安全注意事项

### 19.1 当前安全措施

- 密码哈希保存，不保存明文密码。
- JWT 过期时间控制。
- 所有用户数据查询都校验 `user_id`。
- 前端 401 自动清除 token。
- 后端统一错误码，避免错误格式混乱。

### 19.2 生产化前必须改进

- 更换强随机 `SECRET_KEY`。
- 使用 HTTPS。
- 配置严格 CORS，不允许 `allow_origins=["*"]`。
- token 不建议长期放 localStorage，可考虑 HttpOnly Cookie。
- 增加登录限流。
- 增加请求日志脱敏。
- 增加数据库迁移和备份策略。
- 增加 LLM 输出安全过滤。
- 增加工具调用权限和审计。

## 21. 项目亮点

### 21.1 工程结构清晰

项目不是把所有逻辑写在一个文件里，而是拆成 routers、schemas、services、models、agent 等层，便于维护和扩展。

### 21.2 Agent 与业务系统解耦

Agent 工具不直接写复杂 SQL，而是复用 service 层能力。这样普通 API 和 Agent 工具共享同一套业务规则。

### 21.3 ToolResult 结构化

工具结果不是简单字符串，而是包含 ok、message、data、task_patch、error_code。这样既方便 LLM 回复，也方便状态机和测试。

### 21.4 会话记忆有工程边界

记忆写入放在后台任务中，避免摘要模型影响用户主请求。记忆类型也通过 enum 管理，后续扩展成本低。

### 21.5 前端具备完整用户路径

不是只有接口调试页，而是有登录、聊天、订单、售后、个人资料页面，能够完整演示用户使用流程。

## 22. 维护约定

### 22.1 编码约定

- Python 保持 3.9 兼容。
- 业务逻辑优先放 service 层。
- 路由层只做请求解析、依赖注入和响应返回。
- Agent 工具复用 service 层。
- 错误优先使用 `AppError`。
- 数据归属校验必须放在 service 层。

### 22.2 文档约定

- 数据库结构变更时同步更新 `DATABASE.md` 和本文件。
- Agent 图结构变更时同步更新 `AGENT_ARCH.md` 和本文件。
- API 变更时同步更新本文件第 8 章。
- 新增环境变量时同步更新 `.env.example` 和本文件第 5 章。

### 22.3 测试约定

- 新增 Agent 工具必须加工具测试。
- 修改状态机必须加状态机测试。
- 修改记忆逻辑必须加 memory_service 测试。
- 修改 prompt 规则后建议人工跑核心对话路径。
